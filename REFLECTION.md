# REFLECTION

## Почему эта задача

Я выбрал агента для подбора pet-проектов, потому что это частая и довольно практичная проблема. Человек может уже знать Python, FastAPI, SQL или интересоваться LLM, но всё равно не понимать, какой проект собрать для портфолио. Советы вроде “сделай CRUD” или “сделай таск-трекер” слишком общие: они не объясняют, почему идея подходит именно этому человеку и на какие реальные примеры можно опереться.

Мне хотелось сделать не просто обёртку над LLM, которая сразу выдаёт три идеи. Я хотел, чтобы агент сначала понял ввод пользователя, потом решил, нужны ли инструменты, нашёл open-source референсы и только после этого собрал ответ. Поэтому в проекте есть разбор профиля по локальному каталогу навыков, роутинг инструментов, GitHub Search и Hacker News.

## Где агент сломался

Первое слабое место — локальная LLM. Я использовал Ollama, и маленькая модель не всегда стабильно возвращала JSON. Иногда список идей лежал не в `ideas`, иногда появлялся лишний уровень объекта, иногда формат был почти правильный, но код уже не мог его разобрать. Из-за этого сценарий мог падать не из-за плохой идеи, а просто из-за формы ответа.

Чтобы это сгладить, я сделал более мягкий парсер и fallback. Парсер ищет список идей в нескольких местах, а если нормальный ответ получить не удалось, агент возвращает запасные идеи. Это не идеально, но лучше, чем ошибка в CLI.

Второй неприятный момент — ссылки. LLM может придумать GitHub URL, который выглядит правдоподобно, но не был найден инструментом. Для этого проекта это критично, потому что референсы должны быть настоящими. Поэтому я добавил проверку: ссылка остаётся в ответе только если она пришла из текущего GitHub Search.

Ещё был обычный инженерный разъезд: renderer поменял debug-заголовок, а тест всё ещё ждал старую строку. Плюс settings читали `GITHUB_SEARCH_LIMIT`, но CLI использовал захардкоженный limit. Это мелочи, но они хорошо показывают, что даже маленькому агенту нужны нормальные контракты и тесты.

## Что сделал бы дальше

Сначала я бы дочистил архитектуру, а не добавлял новые фичи. Сейчас часть знаний уже вынесена в `knowledge/registry.toml`, но GitHub ranking, fallback-идеи и правила query expansion всё ещё частично живут прямо в Python-коде. Их лучше вынести в конфиг или отдельный слой знаний.

Потом я бы сделал research-слой асинхронным. Сейчас GitHub и Hacker News вызываются через `requests` и последовательно. Если агент делает несколько GitHub-запросов, задержка быстро растёт. `httpx` и параллельные запросы тут были бы полезнее.

Ещё я бы добавил более строгую валидацию результата через Pydantic: проверять количество идей, обязательные поля, ссылки и пустые fallback-сценарии.

И последнее — режим доработки идеи. Сейчас агент выдаёт три варианта и заканчивает. Было бы полезнее выбрать одну идею и попросить агента разложить её в MVP, задачи на первую неделю, архитектуру и конкретные open-source примеры по фичам.
